# API

The API provides a consistent request flow for social data collection:

```text
submit -> task_id -> task status/results/files
```

## Authentication

```http
Authorization: Bearer YOUR_API_KEY
Content-Type: application/json
```

## Common Submit Body

Input fields are sent directly in the JSON body.

```json
{
  "urls": ["https://www.instagram.com/p/POST_CODE/"],
  "results_limit": 10,
  "callback_url": "https://your-domain.com/webhook"
}
```

`callback_url` is optional. If provided, SocQ sends a POST request after the task reaches a terminal state. See [Callbacks](/api-manual/agent-api/callbacks) for the payload shape.

## Use the same endpoint through MCP or CLI

Every REST collection endpoint has a stable endpoint ID, typed MCP tool, and CLI command. The JSON request fields remain the same:

| Interface | YouTube comments example | Where the JSON body goes |
| --- | --- | --- |
| REST | `POST /v1/youtube/comments` | HTTP request body |
| Compact MCP | `socq_execute`, endpoint `youtube/comments` | Nested `input` object |
| Typed MCP | `socq_youtube_comments` | Top-level tool arguments |
| CLI | `socq youtube comments` | Command flags or `--input-file` |

```text
POST /v1/youtube/comments
          -> endpoint: youtube/comments
          -> tool: socq_youtube_comments
          -> CLI: socq youtube comments
```

Each endpoint page shows its exact mapping. See [MCP quickstart](/integrations/mcp) for a complete submit, poll, paginate, and result example.

## Result Storage

SocQ stores two forms of results:

| Storage | Purpose |
| --- | --- |
| Normalized records | Returned by `/v1/tasks/{task_id}` after the task succeeds |
| Raw JSONL file | Download from `/v1/tasks/{task_id}/files` |

Raw files use this path format:

```text
agent-results/{task_id}/raw_jsonl.jsonl.gz
```

## Billing

Agent APIs are billed by returned result record. The submit request may reserve credits from `results_limit` or the number of submitted URLs/usernames; after the task finishes, SocQ settles the final charge from the actual `result_count` and returns any unused reserved credits. Comments endpoints do not accept `results_limit` and are billed from the actual number of comments returned.
